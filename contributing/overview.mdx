We welcome pull requests from everyone. If you have an idea for a feature or a bug you want to fix, this guide covers everything you need to know to contribute effectively.

<Note>
  Before opening a PR, discuss the change you wish to make via a GitHub issue or another communication channel with the maintainers. This avoids wasted effort and ensures your contribution aligns with the project's direction.
</Note>

## Before you start

### A note on AI-generated PRs

The maintainer explicitly does not accept pull requests that were generated by a coding agent and submitted without genuine understanding of the code. PRs where AI authorship is apparent — or even suspected — will be closed.

If you cannot read Go or are not familiar enough with the lazygit codebase to evaluate the output yourself, that is a prerequisite to fix before contributing a PR, not something to hand off to a model.

### Go is beginner-friendly

If you have never written Go before, you are in good company — lazygit was the maintainer's first Go program, and many contributors came in the same way. Go is widely considered an easy language to pick up, so lazygit is a reasonable place to gain experience with it.

See the [codebase guide](/contributing/codebase-guide) for an orientation to the packages and concepts.

## PR workflow

<Steps>
  <Step title="Fork and create a branch">
    Fork the repository and create a feature branch from `master`. Do **not** open PRs from your fork's `master` branch — maintainers sometimes push changes to your branch during review and cannot do so if you use `master`.

    ```bash
    git checkout -b my-feature master
    ```
  </Step>
  <Step title="Make your changes">
    Write your code, add tests if you have added testable behavior, and update documentation if needed. See the [design principles](#design-principles) below to understand what the maintainers are looking for.
  </Step>
  <Step title="Organize your commits">
    Take time to structure your commit history before opening the PR. See the [commit history](#commit-history) section for the rules.
  </Step>
  <Step title="Open the pull request">
    Push your branch and open a PR against `master`. Write a clear description of what the PR does and why.
  </Step>
  <Step title="Respond to review with fixup commits">
    When reviewers request changes, use [fixup commits](/contributing/overview#fixup-commits) rather than amending or force-pushing a rewritten history. This makes your changes transparent to reviewers while keeping the history clean.
  </Step>
</Steps>

## Commit history

lazygit values a clean and useful commit history. Commits are **not** squashed on merge, so what you submit is what lands in `master`.

<AccordionGroup>
  <Accordion title="Keep refactorings and behavior changes separate">
    Refactorings and behavior changes should live in separate commits. There are very few legitimate exceptions to this rule.
  </Accordion>
  <Accordion title="Strive for minimal, self-contained commits">
    Every change that is independent from other changes should be in its own commit, with a commit message that explains **why** the change is being made — not just what it does. Follow the [good commit message](http://tbaggery.com/2008/04/19/a-note-about-git-commit-messages.html) conventions.
  </Accordion>
  <Accordion title="Use fixup commits during review">
    When iterating during review — fixing a bug a reviewer found, or incorporating requested changes — do not pile new commits on top of your branch history indiscriminately. Use fixup commits so that your changes remain transparent to the reviewer while the eventual history stays clean.

    If you are not familiar with fixup commits, read the [fixup commits guide](/contributing/overview#fixup-commits) below.
  </Accordion>
</AccordionGroup>

## Fixup commits

A **fixup commit** is a commit whose subject begins with `fixup! ` followed by the subject of the commit it is meant to fix up. This convention lets reviewers see exactly what changed in response to their feedback, and lets git automatically squash the fixup into the right place when you are ready to merge.

### Creating a fixup commit in lazygit

In the **Commits** view, select the commit you want to create a fixup for and press `shift-F` ("Create fixup commit for this commit"). lazygit creates a new commit with the correct `fixup! ` prefix automatically.

<Tip>
  Do not confuse `shift-F` (create a fixup commit) with `f` (immediately squash the selected commit into its parent). You want `shift-F` during review.
</Tip>

### Finding the right base commit

When your working copy has changes spread across several commits, press `ctrl-f` in the **Files** view to automatically select the correct base commit in the Commits view. From there, press `shift-F` to create the fixup commit, or `shift-A` to amend directly if the branch has not been published.

### Squashing fixup commits before merge

When you are ready to merge and want to collapse all the fixup commits into their targets, select the first commit of your branch in the Commits view and press `shift-S` ("Squash all `fixup!` commits above selected commit (autosquash)").

## Setting up your development environment

Choose the approach that suits you best.

<Tabs>
  <Tab title="Local setup">
    Install Go (see `go.mod` for the required version), clone the repo, and run lazygit from source:

    ```bash
    git clone https://github.com/jesseduffield/lazygit.git
    cd lazygit
    go run main.go
    ```

    For code formatting, install and configure `gofumpt` (a stricter superset of `gofmt`):

    ```bash
    go install mvdan.cc/gofumpt@latest && gofumpt -l -w .
    ```

    If you use VSCode, enable `gofumpt` in your settings:

    ```jsonc
    // .vscode/settings.json
    {
      "gopls": {
        "formatting.gofumpt": true
      }
    }
    ```
  </Tab>
  <Tab title="VSCode dev container">
    If you want to skip installing Go, extensions, and extra tools yourself, open the repository in a VSCode dev container. This requires:

    - Docker installed and running
    - The Dev Containers extension installed in VSCode

    Open the repository folder in VSCode and choose **Reopen in Container** when prompted. See the [Dev Containers docs](https://code.visualstudio.com/docs/devcontainers/containers) for more detail.
  </Tab>
  <Tab title="GitHub Codespace">
    Fork the repository and open your fork in a GitHub Codespace. The Codespace comes with Go and all required tools pre-installed. To run lazygit from the integrated terminal:

    ```bash
    go run main.go
    ```

    No local installation required.
  </Tab>
  <Tab title="Nix">
    If you use Nix, the repository includes a flake that provides a complete development environment:

    ```bash
    nix develop
    ```

    This drops you into a shell with the Go toolchain, `golangci-lint`, `git`, and `make`. You can also build and run directly:

    ```bash
    # Build
    nix build

    # Run
    nix run
    ```

    The flake supports `x86_64-linux`, `aarch64-linux`, `x86_64-darwin`, and `aarch64-darwin`.
  </Tab>
</Tabs>

## Running tests

lazygit has two kinds of tests.

### Unit tests

Unit tests live in `_test.go` files alongside the code they test and are run with the standard Go test command:

```bash
go test ./...
```

### Integration tests

Integration tests run a real lazygit session driven by a scripted robot and assert that everything behaves as expected. Tests live in `pkg/integration/tests/`.

The easiest way to run integration tests is via the TUI runner:

```bash
go run cmd/integration_test/main.go tui
```

You can also run tests from the command line directly:

```bash
# Run all tests
go run cmd/integration_test/main.go cli

# Run a specific test (name is based on its path)
go run cmd/integration_test/main.go cli commit/new_branch

# Run a test slowly so you can watch what it does
go run cmd/integration_test/main.go cli --slow commit/new_branch
```

<Tip>
  Use **sandbox mode** to run a test's setup step and then drive lazygit yourself, without the robot taking over. This is a great way to reproduce an edge case without manually setting up repo state. Press `s` on a test in the TUI, or pass `--sandbox` on the CLI.
</Tip>

See the [codebase guide](/contributing/codebase-guide#integration-tests) for detail on writing new integration tests.

## Debugging

Open two terminal tabs: one to run lazygit in debug mode, and one to stream its logs.

**Terminal 1 — run lazygit:**

```bash
go run main.go --debug
```

**Terminal 2 — stream logs:**

```bash
lazygit --logs
```

From most places in the codebase you can write to the logger:

```go
self.c.Log.Warn("something happened")
```

To reduce noise from existing log lines, set the log level:

```bash
LOG_LEVEL=warn go run main.go --debug
```

### VSCode debugger

To trigger a debug session from VSCode, add the following launch configuration. The `externalTerminal` console setting is required so that the lazygit TUI can render in its own window while the debugger is attached.

```jsonc
// .vscode/launch.json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "debug lazygit",
      "type": "go",
      "request": "launch",
      "mode": "auto",
      "program": "main.go",
      "args": ["--debug"],
      "console": "externalTerminal"
    }
  ]
}
```

### Logging from vendor code

If you need to log from code inside the `vendor/` directory (for example, the `gocui` package), use the global logger instead:

```bash
LAZYGIT_LOG_PATH=/tmp/lazygit.log go run main.go --debug
```

```go
import "github.com/jesseduffield/lazygit/pkg/logs"

logs.Global.Warn("something from vendor code")
```

## Go conventions in this project

lazygit follows standard Go idioms with a few intentional deviations.

<AccordionGroup>
  <Accordion title="Use `self` as the receiver name">
    There is no shame in using `self` as the receiver name in struct methods — lazygit encourages it.
  </Accordion>
  <Accordion title="Prefix interfaces with `I` when they have multiple methods">
    Instead of suffixing with `er` (the standard Go convention), interfaces with several methods are prefixed with `I`. For example, `IController` rather than `Controller` suffixed with `er`.
  </Accordion>
  <Accordion title="Make interface satisfaction explicit">
    When a struct is meant to implement an interface, make that explicit at the top of the file:

    ```go
    var _ MyInterface = &MyStruct{}
    ```

    This makes intent clear and produces a compile error in the right file if the interface is not satisfied.
  </Accordion>
</AccordionGroup>

## Internationalisation

All user-facing strings are defined in `pkg/i18n/english.go`. If you are adding new text, add a field to the `TranslationSet` struct and provide the English string in the `EnglishTranslationSet()` function. Access strings via `self.c.Tr.YourNewField`.

Use sentence case for all strings ("Stage this file", not "Stage This File").

Translations are managed through [Crowdin](https://crowdin.com/project/lazygit/). If you want to contribute translations, join the project there — do not edit the files in `pkg/i18n/translations/` directly.

## Updating dependencies

### Updating gocui

[gocui](https://github.com/jesseduffield/gocui) is the underlying library that handles the event loop, keypresses, and rendering. To update it:

<Steps>
  <Step title="Make changes in vendor/">
    Make your changes to gocui inside `vendor/github.com/jesseduffield/gocui` so you can test them against lazygit immediately.
  </Step>
  <Step title="Copy changes to the gocui repo">
    Copy your changes to a local clone of the gocui repository. Use the `awesome` branch, not `master`.
  </Step>
  <Step title="Open a PR on gocui">
    Raise a PR on the gocui repository with your changes.
  </Step>
  <Step title="Bump the version in lazygit">
    After the gocui PR is merged, bump the version in lazygit:

    ```bash
    ./scripts/bump_gocui.sh
    ```
  </Step>
  <Step title="Open a PR on lazygit">
    Raise a PR in lazygit with the version bump.
  </Step>
</Steps>

### Updating lazycore

[lazycore](https://github.com/jesseduffield/lazycore) contains functionality shared between lazygit and lazydocker. The process is the same as for gocui, except use the `master` branch and run:

```bash
./scripts/bump_lazycore.sh
```

## Design principles

When building a feature or proposing a change, consider lazygit's design principles. Many of these are in tension with each other — that is intentional, and requires judgement.

<CardGroup cols={2}>
  <Card title="Discoverability" icon="compass">
    TUIs are notoriously hard to learn. Use tooltips, visual cues, and contextual keybinding prompts to make lazygit easy to grok without reading a manual. If a keybinding is disabled, explain why.
  </Card>
  <Card title="Simplicity" icon="minimize">
    Most git use cases are simple. Keep the most common workflows dead simple. Use sensible defaults. Think hard before adding a new configuration option — there are already too many.
  </Card>
  <Card title="Safety" icon="shield">
    Git mistakes are easy to make and painful to undo. Prompt for confirmation before anything hard to reverse. Make it easy to correct mistakes via the undo action (`z`) and the escape key.
  </Card>
  <Card title="Power" icon="bolt">
    Users should rarely need to drop to the git CLI. Support complex but common workflows (interactive rebase, custom patches). Use the custom commands system for rare edge cases.
  </Card>
  <Card title="Speed" icon="gauge">
    Pro users need to move fast. Minimize keypresses. Keep startup fast — if something is slow, make it non-blocking. Support muscle memory: prefer disabling menu items to hiding them.
  </Card>
  <Card title="Conformity with git" icon="code-branch">
    Honour the git config. Work with git, not against it. Avoid storing lazygit-specific state that could live in git. Only override git's defaults when there is a well-considered reason.
  </Card>
</CardGroup>

<Note>
  When principles conflict — and they often do — err on the side of safety and simplicity as the default, while allowing power users to opt in to faster or more powerful behavior via configuration or separate keybindings.
</Note>
